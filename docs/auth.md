# 会员鉴权方案

对于需要会员才能使用的技能，有几种防倒卖方案：

## 方案一：后端 API 鉴权（推荐，防盗版最强）

### 原理
Skill 代码本身可以公开下载，但每次调用都需要向你的服务器验证会员 token。

### 实现
1. 用户订阅后，生成唯一的 `user_token` 发给用户
2. 用户在 OpenClaw 配置文件中设置环境变量：`export MY_SKILLHUB_TOKEN=xxx`
3. Skill 调用时，自动携带 token 请求你的服务器：
```javascript
const token = process.env.MY_SKILLHUB_TOKEN;
const response = await fetch('https://your-domain.com/api/verify', {
  method: 'POST',
  body: JSON.stringify({ token })
});
if (!response.ok) {
  throw new Error('会员验证失败，请检查订阅状态');
}
```
4. 你的服务器校验 token 是否有效（是否订阅、是否过期）

### 优点
- 完全防止倒卖，没有有效 token 就算拿到代码也用不了
- 可以统计调用次数
- 随时可以吊销权限

### 缺点
- 需要你维护一个简单的后端服务

---

## 方案二：私有仓库 + 授权访问（中等）

### 原理
会员技能代码放在私有 Git 仓库，只有授权用户才能访问。

### 实现
1. 用户订阅后，将用户的 GitHub 账号添加到仓库合作者
2. Skill 的 repository 地址指向你的私有仓库
3. OpenClaw 安装时需要用户认证

### 优点
- 实现简单，利用 GitHub 已有权限系统
- 不需要自己写后端

### 缺点
- 用户拿到仓库权限后还是可以 clone 下来分享出去
- 管理授权比较麻烦

---

## 方案三：水印追踪 + 社区监督（简单）

### 原理
每个用户下载的 Skill 包中嵌入唯一水印，发现倒卖给可以追踪来源。

### 实现
1. 下载时根据用户 ID 在 SKILL.md 和代码中插入隐形水印
2. 在 SKILL.md 开头声明："此技能受版权保护，仅授权给 XXX 使用"
3. 发现公开倒卖后，列入黑名单不再更新

### 优点
- 实现最简单
- 对小流量来说足够用

### 缺点
- 不能防止倒卖，只能追溯
- 适合早期规模小的时候用

---

## 推荐起步顺序

1. **早期**：先用方案三，成本最低，验证市场
2. **有一定会员量后**：改用方案一，对会员专享的核心技能做后端鉴权
3. **不推荐**：方案二，体验不好而且防护弱
